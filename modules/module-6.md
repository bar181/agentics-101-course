## ***Lesson 6.1: Enhance Your Current Agent***

***Time:** 10 minutes*  
 ***Outcome:** Scale using multi-file specifications*

### ***Lesson Overview***

*Your agent works for one spec. Let's make it handle multiple specs at once. This is how you build complex projects‚Äîmany small specifications combined, not one giant prompt.*

### ***The Multi-File Method***

*Instead of one large `page-spec.md`, create separate specs:*

```
specs/
‚îú‚îÄ‚îÄ header-spec.md    (Navigation bar)
‚îú‚îÄ‚îÄ content-spec.md   (Main content)
‚îú‚îÄ‚îÄ footer-spec.md    (Footer section)
‚îî‚îÄ‚îÄ style-spec.md     (Color theme)
```

***How to use multiple specs:***

1. *Open Claude.ai*  
2. ***Drag and drop all spec files** into the chat*  
3. *Prompt: "Read these specs and generate a complete index.html file that implements all of them"*

*Claude reads all files and combines them into cohesive code.*

### ***Quick Enhancements to Try***

***Enhancement 1: Add a Feature***

*Create `specs/contact-form-spec.md` with form requirements. Feed it to Claude with your existing spec: "Add this contact form to the existing page following both specifications."*

***Enhancement 2: Create Theme Variants***

*Keep the structure, change the styling. Create `specs/dark-theme-spec.md`: "Apply this dark theme to the existing structure without changing the HTML layout."*

***Enhancement 3: Version Your Specs***

*Don't delete old specs. Version them:*

```
specs/
‚îú‚îÄ‚îÄ v1-basic.md       (MVP)
‚îú‚îÄ‚îÄ v2-enhanced.md    (Added features)
‚îî‚îÄ‚îÄ v3-polished.md    (Final version)
```

*Show Claude the progression: "Here's v1. Now upgrade using v2 requirements."*

***Try one now:** Pick an enhancement and implement it in 5 minutes.*

### ***Choose Your Next Agent***

***Option 1: Documentation Agent üõ°Ô∏è** (Safest)*

 ***Input:** Your `index.html` file*

 ***Why start here:** Turns spaghetti code into professional code*

## ***üíé Optional: What's Next in Your Journey***

***For students ready to explore beyond the basics***

### ***Professional Tools Preview***

*You've mastered the manual workflow (copying code from Claude to GitHub). Professional agentic engineers use tools that automate this process. These tools aren't better or worse‚Äîthey're faster once you understand the fundamentals.*

***Three Tools to Explore:***

***Cursor***

* *VS Code fork built for AI development*  
* *Press Cmd/Ctrl+I, describe what you want, AI writes code directly*  
* *Free tier available*  
* *Best for: Smooth, integrated experience*

***Roo Code***

* *VS Code extension*  
* *Works with multiple AI providers (Claude, GPT-4, Gemini)*  
* *Free extension (requires API keys)*  
* *Best for: Flexibility and customization*

***Claude Code***

* *Command-line tool*  
* *Type commands in terminal, AI modifies files*  
* *Free tool (pay-as-you-go API usage)*  
* *Best for: Terminal-comfortable developers*

### ***When to Adopt Pro Tools***

***Adopt automated tools when:***

* *‚úÖ You've built 3+ projects manually*  
* *‚úÖ You can write specs without examples*  
* *‚úÖ You spot code errors quickly during review*  
* *‚úÖ You understand what every workflow step does*  
* *‚úÖ You've memorized the Three Laws*

***Don't rush to automation if:***

* *‚ùå You're still learning core concepts*  
* *‚ùå Manual workflow feels uncomfortable*  
* *‚ùå You can't explain the process to someone else*  
* *‚ùå You haven't completed your 30-day plan*

***Why wait?** Manual workflow builds intuition that automated tools rely on. Sarah used manual workflow for three months before adopting Cursor. She said: "Because I'd copied code 100+ times manually, I could review diffs in 5 seconds. Automation made me faster, not better‚Äîthe manual practice made me better."*

### ***Safety Reminder for Automated Tools***

*If you explore these tools, the Three Laws become even more critical:*

***Law 1: Human Approval Required***

* *These tools can write directly to files*  
* *ALWAYS review diffs before accepting changes*  
* *Never enable "auto-approve" features*

***Law 2: Bounded Scope***

* *Explicitly limit which folders tools can modify*  
* *Keep specs/ read-only, docs/ writable*  
* *Protect .git directory from changes*

***Law 3: Read Before Write***

* *Show tools current file state*  
* *Provide context before requesting changes*  
* *Don't let tools make blind modifications*

***Recovery commands** (if tools break something):*

```shell
git diff              # See what changed
git checkout -- file  # Undo uncommitted changes
git reset HEAD~1      # Undo last commit
```
